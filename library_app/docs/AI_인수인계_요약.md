# AI 인수인계 요약

## 1. 프로젝트 현재 방향

이 프로젝트는 공공도서관 장서 데이터를 분석해 사서가 장서 운영 의사결정을 할 수 있도록 돕는 대시보드이다.

최종 방향은 단순 시각화 대시보드가 아니라 다음에 가깝다.

> 공공도서관 장서·대출 데이터를 활용해 장서 공급과 이용 수요의 불균형을 통계적으로 진단하고, 수서·복본·정리 우선순위를 제안하는 사서용 자동 장서 진단 대시보드

핵심은 특정 두 도서관 비교가 아니다. 어떤 도서관 데이터가 들어와도 같은 진단 엔진으로 분석할 수 있는 범용 장서 진단 구조를 지향한다.

## 2. 저장소 구조

현재 저장소는 두 앱으로 분리되어 있다.

```text
사이드프로젝트/
├─ generic_dashboard/      # 범용 데이터 분석 대시보드
│  ├─ app.py
│  ├─ pages/
│  └─ helpers/
├─ library_app/            # 도서관 공모전용 장서 진단 대시보드
│  ├─ app.py
│  ├─ data/
│  ├─ docs/
│  ├─ loader.py
│  ├─ diagnosis.py
│  ├─ metrics.py
│  └─ prescribe.py
├─ requirements.txt
├─ README.md
├─ 실행.bat
└─ 실행_도서관.bat
```

`generic_dashboard/`는 기존 범용 분석 앱이다. CSV, Excel, JSON 업로드 후 품질 점검, 시각화, 통계분석, 회귀분석, 보고서 생성을 하는 구조이다.

`library_app/`가 앞으로 집중할 공모전용 앱이다. 도서관 장서·대출 데이터 기반 진단, 통계 분석, 처방 로직이 들어갈 곳이다.

## 4. 작년 대상팀 분석에서 얻은 참고점

확인한 작년 대상팀 관련 링크:

- LIB analysis: https://libanalysisv2-u6kzbhl4x2npgacbuplxyx.streamlit.app/
- LIBanalysis GitHub: https://github.com/FiveEastRoot/LIBanalysis_v2
- IBanalysis GitHub: https://github.com/FiveEastRoot/IBanalysis_rep

분석 결과:

- `LIBscope_v2` 계열은 API 우선, 정적 데이터 fallback, 캐시 전략을 사용한 것으로 보인다.
- `LIBanalysis_v2` 계열은 설문 엑셀 파일 업로드 기반 앱이다.
- `LIBanalysis_v2` GitHub에는 설문 원자료 파일이 없다. `assets/`에는 `logo.png`만 있다.
- `app.py`에서 `st.file_uploader(".xlsx")`와 `pd.read_excel(uploaded)`로 사용자가 파일을 올리는 구조이다.
- 코드 안에는 `Q1~Q9`, `SQ`, `DQ`, `Q9-DS-4` 등 설문 문항 코드가 구체적으로 들어 있다.
- 따라서 작년팀이 실제 설문을 직접 했는지는 확정할 수 없지만, 특정 설문 응답 양식 또는 가공된 설문 데이터에 맞춘 앱을 만든 것으로 보인다.

가져올 점:

- 도서관 선택형 대시보드
- 기본 분석 -> 심화 분석 -> 전략 인사이트 흐름
- API 우선 + 정적 fallback + 캐시 구조
- AI는 분석 판단이 아니라 해석 리포트 보조 기능으로 사용

버릴 점:

- 설문 데이터 중심 구조
- 단순 업로드형 분석에 머무르는 방식
- 특정 지역/특정 데이터에만 종속된 화면
- AI가 통계 판단을 대신하는 구조

```text
C:\Users\itwill\Downloads\서울시민의 도서관 이용실태\(원데이터)도서관 이용실태조사 DATA.xlsx
C:\Users\itwill\Downloads\서울시민의 도서관 이용실태\(원데이터)도서관 이용실태조사 코드북.xlsx
```

확인 내용:

- `DATA.xlsx`는 응답자 약 3064명, 컬럼 약 456개 구조이다.
- 주요 컬럼은 `SQ1`, `SQ2`, `SQ3`, `Q1A1`, `Q1A2`, `Q9`, `DQ1`, `DQ2`, `DQ3`, `DQ4` 등이다.
- 설문지 본문은 `2018 서울시민 도서관 이용현황 조사`로 보인다.
- 작년팀 코드의 핵심 문항인 `Q9-DS-4`는 해당 데이터에서 확인되지 않았다.
- 작년팀 코드의 `Q1-1` 형식과 다르게 이 데이터는 `Q1A1` 형식이다.

결론:

- 사용자가 찾은 설문은 작년팀 데이터와 비슷한 계열이지만, 그대로 같은 원본일 가능성은 낮다.
- 다만 공개 설문 원데이터를 활용하면 직접 설문을 하지 않고도 설문 분석 앱을 만들 수 있다는 참고 사례가 된다.
- 우리 프로젝트의 핵심은 장서·대출 데이터이므로 설문 데이터는 메인보다는 보조 참고 데이터로 두는 편이 낫다.

## 6. 핵심 분석 개념

우리 프로젝트의 핵심 지표는 장서 공급과 이용 수요의 차이다.

```text
공급비중 = 특정 주제 도서권수 / 전체 도서권수
수요비중 = 특정 주제 대출건수 / 전체 대출건수
수급격차 = 수요비중 - 공급비중
대출률 = 대출건수 / 도서권수
```

해석 예시:

```text
수급격차가 크고 양수
→ 책은 상대적으로 적은데 대출은 많은 분야
→ 수서 확충 후보

대출률 높음 + 도서권수 적음
→ 현재 보유 권수에 비해 이용이 몰리는 분야
→ 복본 검토 후보

수급격차 음수 + 대출률 낮음 + 장서연령 높음
→ 책은 많지만 잘 이용되지 않는 분야
→ 정리/점검 후보

수급격차 0 근처
→ 장서 공급과 이용 수요가 비교적 균형
→ 유지 후보
```

## 7. 사서용 화면 언어

통계 용어를 전면에 내세우면 안 된다. 사서가 해결하고 싶은 업무 질문 중심으로 보여줘야 한다.

권장 화면 문구:

```text
무엇을 확인하고 싶나요?

[책을 더 사야 할 분야]
책은 적지만 대출이 많은 주제를 찾습니다.

[복본이 필요한 분야]
보유 권수에 비해 이용이 몰리는 주제를 찾습니다.

[이용이 낮은 장서]
책은 많지만 잘 빌려지지 않는 분야를 찾습니다.

[장서와 이용 수요 비교]
주제별 장서 비율과 대출 비율이 맞는지 확인합니다.
```

통계 용어 변환:

```text
종속변수 -> 확인하고 싶은 결과
독립변수 -> 영향을 줄 수 있는 요인
EDA -> 데이터에서 단서 찾기
통계 검정 -> 우연인지 확인하기
모델링 -> 여러 요인을 종합해 예측/판단하기
```

## 8. EDA 방향

현재 범용 대시보드는 시각화와 통계 기능이 나열된 구조에 가깝다.

앞으로는 다음 흐름으로 바꾸는 것이 좋다.

```text
품질 점검
→ 변수 후보 탐색
→ 확인하고 싶은 결과 선택
→ 결과와 관련 있는 변수 1차 스크리닝
→ 가설 생성
→ 통계 검정
→ 모델링
→ 해석
```

핵심 질문:

> 이 변수는 내가 설명하고 싶은 결과값과 관련이 있는가?

도서관 앱에서는 예를 들어 다음처럼 사용할 수 있다.

```text
확인하고 싶은 결과: 대출률
후보 요인: 도서권수, KDC 주제분류, 출판년도, 등록연도, 장서연령

확인하고 싶은 결과: 수급격차
후보 요인: KDC 주제분류, 도서권수, 대출건수, 출판년도
```

종속변수 유형별 자동화:

```text
결과가 숫자형
→ 숫자형 후보 요인: 상관분석, 산점도
→ 범주형 후보 요인: 그룹별 평균 비교, 박스플롯, ANOVA/Kruskal
→ 날짜형 후보 요인: 기간별 추세
→ 모델링: 회귀분석

결과가 범주형
→ 숫자형 후보 요인: 그룹별 분포 비교, t-test/ANOVA
→ 범주형 후보 요인: 교차표, 카이제곱 검정
→ 모델링: 분류 모델

결과가 이진형
→ 숫자형 후보 요인: 두 그룹 평균 비교
→ 범주형 후보 요인: 교차표, 카이제곱 검정
→ 모델링: 로지스틱 회귀 또는 분류 모델
```

사용자에게는 검정법 이름보다 쉬운 결과를 보여준다.

```text
이 변수는 결과와 관련이 있어 보입니다.
근거: 그룹별 평균 차이가 큽니다.
통계 확인 결과: 우연으로 보기 어려운 차이입니다.
추천: 모델 후보 변수로 포함해도 좋습니다.
```

## 9. 자동 진단 가능성

데이터 형식이 정해져 있다면 자동 분석은 가능하다.

예상 필수 컬럼:

```text
도서명
ISBN
KDC 주제분류번호
도서권수
대출건수
출판년도
등록일자
```

자동 생성 가능한 파생변수:

```text
대출률 = 대출건수 / 도서권수
공급비중 = 주제별 도서권수 / 전체 도서권수
수요비중 = 주제별 대출건수 / 전체 대출건수
수급격차 = 수요비중 - 공급비중
장서연령 = 현재연도 - 출판년도
등록후경과연수 = 현재연도 - 등록연도
미대출여부 = 대출건수 == 0
KDC대분류 = 주제분류번호 첫 자리
```

자동 분석 흐름:

```text
데이터 입력
→ 컬럼 검증
→ 파생변수 생성
→ 기본 EDA
→ 변수 스크리닝
→ 통계 검정
→ 모델링
→ 처방 결과 생성
→ 사서용 해석 리포트
```

## 10. 도서관 선택형 자동 진단

궁극적으로는 사서가 도서관만 선택하면 자동으로 진단이 돌아가는 구조를 목표로 한다.

예상 흐름:

```text
도서관 선택
→ API 또는 샘플/캐시 데이터 로드
→ 자동 장서 진단
→ 확충/복본/정리 후보 출력
→ 사서용 해석 문장 출력
```

도서관정보나루 API를 활용하면 다음 구조가 가능하다.

```text
libSrch
→ 도서관 목록 조회

itemSrch
→ 선택 도서관 장서 데이터 조회

loanItemSrchByLib
→ 선택 도서관 인기대출도서/대출횟수 조회
```

다만 API만으로 전체 도서별 대출건수를 충분히 받을 수 있는지는 확인이 필요하다.

따라서 추천 설계는 다음 두 모드이다.

```text
간편 진단 모드
- 사서가 도서관만 선택
- API 기반 자동 분석
- 장서 구성, 인기대출 기반 수요, 부족/과잉 분야 제안

정밀 진단 모드
- 장서·대출 원자료 CSV 업로드
- 도서별 대출건수 기반 정확한 분석
- 수서/복본/정리 우선순위 산출
```

## 12. 팀원 업무분장 제안

팀원이 사용자 포함 4명이라고 가정한다.

```text
A: 데이터 담당
- 샘플 데이터 구조 확인
- 필수 컬럼 정리
- 결측치/이상치 기준 정리
- 업로드 데이터 검증 규칙 설계

B: 통계 분석 담당
- 공급비중, 수요비중, 수급격차 공식 정리
- 대출률, 회전율, 쏠림 지표 정리
- 카이제곱, 상관분석, 회귀분석 적용 가능성 검토

C: 대시보드 담당
- Streamlit 화면 구성
- 기본 분석, 심화 분석, 전략 처방 화면 설계
- 그래프, 표, 필터 구성

사용자: PM + 통합 담당
- 전체 주제와 발표 스토리 정리
- 코드 구조 관리
- 팀원 결과물 통합
- 처방 기준 설계
- 기획서/README/발표자료 정리
```

페이지별 분담보다 데이터/분석/시각화/통합 기준으로 나누는 것이 좋다. 그래야 병렬 작업이 가능하고 계산 기준이 흔들리지 않는다.

## 16. 2026-06-30 작업 업데이트 (MVP 설계 · 로지스틱 · API 확인)

이날 세션에서 MVP 설계를 구체화하고, 수업 트랙(로지스틱)을 실데이터로 검증하고,
대시보드를 선택형 + 전국 지도로 확장했다. 핵심 결정과 사실을 아래에 남긴다.

### 16.1 전략 확정 — "한 몸통, 두 얼굴"

같은 정제 데이터에서 **두 산출물**을 동시에 뽑는다(둘 중 하나만 하면 한쪽 채점이 빈다).

- **① 공모전 데모 트랙**: 집계 기반 자동 진단(수급격차 → 처방 카드) + 선택형 + 전국 지도. 거의 완성.
- **② 수업/분석 트랙**: 미대출 로지스틱 회귀. 산출물 = 분석 노트북.

회의용 정리는 `library_app/docs/내일_회의_브리프.md` (전략 그림 + 1차 MVP 경계 IN/OUT + 확정 안건 9개 + 검증 결과).

### 16.2 선택형 전환 (완료)

- 사이드바: `사전 적재 도서관 선택 → 2.28 / 독산` 드롭다운 + 업로드(보조).
- 구현: `loader.list_sample_libraries()`(data/의 *장서 대출목록* 파일을 {표시명:경로}로),
  `loader.load_sample_collection(path)`(경로 받아 로드, path=None이면 기존 동작).
- 도서관을 바꾸면 같은 엔진이 다른 진단을 냄 = 범용성 데모.

### 16.3 ② 미대출 로지스틱 — 검증 완료 (수업 트랙의 핵심)

- 종속변수 Y = `미대출 = 대출건수 == 0`. 균형: 2.28은 34%(교과서급), 독산은 12.6%(불균형이나 사용 가능).
- 독립변수 X: 장서연령(2026−발행년도), 등록경과(2026−등록연도), 분야(KDC 첫자리), 독자대상(부가기호 첫자리).
  도서권수는 98%가 1권 → 분산 없어 제외.
- 수렴 안정화: **연속변수 표준화** + 희귀범주(50권 미만) '기타' 통합. 학습/검증 7:3 층화분리.
- 결과: 두 도서관 모두 수렴, Test AUC 2.28=0.766 / 독산=0.788, Pseudo R² 0.18/0.21.
- **핵심 발견(정직하게 기록)**: `등록경과`는 두 도서관에서 **공통 신호**(둘 다 OR<1, 신착 함정 통제).
  `장서연령`은 **도서관별로 다름**(2.28 OR 1.38 vs 독산 0.93). 후자는 모델 흠이 아니라
  자기참조 모델 예측("도서관마다 진단이 달라야")과 부합 = 데이터 증거.
- 노트북: `library_app/notebooks/미대출_로지스틱_분석.ipynb` (18셀, 전체 실행 검증됨).

### 16.6 도서관정보나루 API 확인 결과 (중요)

- 우리 분석 재료 = 책별 `도서권수`+`대출건수`+KDC. 이걸 주는 건 usageTrend도 libSrch도 아닌 **itemSrch**.
  - `usageTrend`(요일별/시간대별): 운영 질문(언제 붐비나). 우리 주제와 결이 다른 곁가지 → 1차 제외.
  - `libSrch`(통합정보): libCode·위경도 = 선택형의 **열쇠**(단, 분석 내용 아님). 이미 참여도서관목록 파일로 로컬 보유 → API 불필요.
- **itemSrch 명세의 함정**: 제공 필드로 "도서명·저자·출판사·출판년도·ISBN·KDC·등록일"만 명시 →
  **`대출건수`·`도서권수`가 목록에 없음.** 제목은 "장서/**대출** 조회"인데 필드엔 대출수치가 없어 모순.
- **전 비전을 가르는 단 하나의 질문**: *itemSrch가 책별 대출건수를 주는가?*
  - 주면 → 도서관만 골라도 전국 자동진단(비전 완성). 안 주면 → "간편(API 장서)/정밀(CSV 업로드)" 2모드가 정답.
  - 어느 쪽이든 프로젝트는 안 망함(CSV엔 대출건수 다 있음). 이건 *비전 크기* 결정이지 *가능 여부* 아님.
- **현재 상태**: 발급한 오픈API 인증키가 **승인 대기**(모든 엔드포인트가 `vitalizationErr: API 활성화 상태가 아님`).
  키 활성화되면 itemSrch를 호출해 대출건수 포함 여부를 즉시 확정할 것. (인증키 값은 보안상 이 문서에 남기지 않음.)
- 확인된 libCode: **2.28도서관 = 127058**, 금천구립독산도서관 = 111040 (참여도서관목록 기준).

### 16.7 설계 원칙 결정 — EDA 순서 vs 사서 목적

- **대시보드 메뉴를 EDA 순서(품질점검→스크리닝→검정→모델링)로 만들지 않는다.** 그건 분석가 동선이지 사서 동선이 아니며,
  7장의 "통계 용어 전면" 함정. EDA 순서의 집은 **노트북**(수업 산출물).
- 대시보드 메뉴는 **사서 목적 순서**(장서·수요 맞나 / 정리할 책 / 더 살 책 / 복본). EDA 단계는 각 화면의 접이식
  "분석 근거"(`show_analysis_basis`)로 녹인다. "분석 과정" 신뢰용 페이지 1개는 선택적(곁가지).

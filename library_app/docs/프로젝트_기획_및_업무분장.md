# 프로젝트 기획 · 업무분장 (데이터분석 수업 + 공모전)

> 이 문서는 팀이 합의한 **주제·전략·역할**을 한곳에 모은 기준 문서다.
> 두 목표 — **데이터분석 수업 과제**와 **공모전 출품** — 을 *동순위*로 둔다.

---

## 0. 전제 (두 목표, 동급)

| | 데이터분석 수업 | 공모전 (부문 ①) |
|---|---|---|
| 채점 핵심 | 분석 *과정*을 제대로·설명 가능하게 했나 (전처리·EDA·통계·해석) | 현안을 *해결*하나, 전국으로 *확장*되나 |
| 주 산출물 | 분석 노트북 + 보고서 | 제안서 + 작동 데모(대시보드) |
| 공통 재료 | ← **같은 분석 결과에서 뽑아 씀** → | |

**원칙: 한 몸통, 두 얼굴.** 분석은 한 번 돌리고, 수업엔 "과정·통계"를 두껍게, 공모전엔 "문제→모델→처방→확장"을 두껍게 뽑는다.

## 1. 주제 (확정)

> **"아무 도서관의 대출목록만 넣으면, 그 도서관에 맞는 장서 진단과 수서·정리 처방을 자동으로 내주는 모델"**
>
> 핵심 차별점 = **자기참조 수급격차**: 외부 절대기준이 아니라 *그 도서관 내부의 공급(장서 구성) vs 수요(실제 대출)* 를 비교한다.

- **공급(供給)** = `도서권수` 분포 → 장서가 어떻게 구성돼 있나
- **수요(需要)** = `대출건수` 분포 → 실제로 무엇이 대출되나
- **격차 = 수요비중 − 공급비중** 이 곧 의사결정 신호
  - 수요 > 공급 → **보강(수서) 후보**
  - 공급 > 수요 → **정리/큐레이션 후보**

### 왜 이 주제인가
- 절대기준(예: "청소년 장서 10% 이상")은 도서관마다 다른 역할을 무시해 오진을 낳는다. 자기참조는 도서관 종류가 달라도 공정하게 작동한다.
- 데이터분석 수업용으로도 최적: 전처리·파생변수·지표설계·비교·통계검정까지 *단계마다 보여줄 거리*가 있다.

## 2. 데이터 (사례는 검증용, 핵심은 범용 엔진)

이 프로젝트의 핵심은 **특정 두 도서관을 비교하는 것**이 아니다. 핵심은 표준 장서·대출 데이터가 주어졌을 때 **어느 공공도서관에도 적용 가능한 자동 진단 엔진**을 만드는 것이다.

따라서 아래 실파일은 "비교 대상"이 아니라, 모델이 실제 데이터에서도 작동하는지 확인하는 **검증 사례**다.

| 검증 사례 | 파일 | 규모 | 검증 포인트 |
|---|---|---|---|
| 대구 2·28도서관 | `2.28도서관 장서 대출목록 (2026년 05월).csv` | 약 5.8만권 | 엔진이 실제 장서·대출목록에서 수급격차를 계산하는지 |
| 금천구립 독산도서관 | `금천구립독산도서관 장서 대출목록 (2026년 05월).csv` | 13.3만종 / 13.6만권, 대출 143.6만건 | 같은 엔진이 다른 데이터에서도 다른 진단·처방을 내는지 |

> **대구 vs 금천이 주제가 아니다.**
> 두 사례는 "왜 이 둘인가?"를 설명하기 위한 비교 대상이 아니라, "데이터가 달라도 같은 진단 엔진이 돌아가는가?"를 보여주는 데모 재료다.

공통 컬럼(표준 data4library 포맷): `도서명·저자·출판사·발행년도·ISBN·세트ISBN·부가기호·권·주제분류번호·도서권수·대출건수·등록일자`

사례 데이터 선택 원칙:
- **주제와 직접 관련된 컬럼**(`도서권수`, `대출건수`, `주제분류번호`, `ISBN`, `부가기호`)이 있어야 한다.
- 도서관 간 비교는 순위나 우열을 매기기 위한 것이 아니라, **엔진의 범용성 검증**을 위한 것이다.
- 최종 발표에서는 "두 도서관 비교"보다 **업로드/선택된 도서관마다 자동 진단되는 구조**를 먼저 보여준다.
- 사례는 필요하면 바꿀 수 있다. 데이터가 추가되면 같은 엔진에 넣어 결과만 갱신한다.

### 클라우드 배포용 데이터 운영 전략

서비스 형태는 **파일 업로드형 분석기**가 아니라, 기본적으로 **선택형 진단 대시보드**로 설계한다.

> 사용자가 대용량 원천 파일을 매번 올리는 구조가 아니라, 서비스 내부에 구축된 공공데이터를 바탕으로 `지역 → 도서관 → 기간/기준`을 선택하면 진단 결과가 자동으로 바뀌는 구조.

이유:
- 클라우드 배포 환경에서는 파일 업로드 용량 제한과 처리 시간이 병목이 될 수 있다.
- 공모전 관점에서는 "파일을 올려 분석하는 도구"보다 "도서관을 선택하면 바로 진단되는 서비스"가 더 완성도 있게 보인다.
- 대시보드 시연에서 지역/도서관 선택에 따라 결과가 바뀌면 전국 확장성과 활용 장면을 더 쉽게 보여줄 수 있다.

작년 대상팀 `LIBscope_v2`에서 참고할 핵심은 **API 우선 + 정적 데이터 fallback + 캐시** 구조다.

> 가능한 데이터는 API로 최신화하고, API가 실패하거나 대용량 원천 데이터가 필요한 경우에는 저장된 CSV/JSON/대표 데이터를 사용한다. 분석 결과는 캐시해 속도를 확보한다.

우리 프로젝트도 데이터를 전부 같은 방식으로 저장하지 않고, 용도별로 나눈다.

| 데이터 층 | 저장 전략 | 용도 |
|---|---|---|
| 전국 공공도서관 기초 정보 | 사전 적재 | 지역·도서관 선택, 지도/목록, 도서관 코드 연결 |
| 전국 인기대출·급상승·키워드·신착 도서 | API 우선 + 최근 결과 캐시 | 최신 수요 트렌드, 수서 추천 후보 |
| 대표 도서관 장서·대출 원천 데이터 | 우선 탑재 | 데모와 분석 검증. 대구 2·28, 금천 독산 등 |
| 전국 개별 도서관 장서·대출 원천 데이터 | 전량 탑재하지 않음 | 필요 시 API/저장소에서 호출하고, 분석 결과만 캐시 |
| API 실패/제한 상황 | 정적 CSV/JSON fallback | 시연 안정성, 클라우드 장애 대응 |
| 사용자 업로드 파일 | 보조 기능 | 수업 검증, 신규 도서관 실험, API 미연동 데이터 분석 |

따라서 최종 구조는 다음처럼 정리한다.

> **평소에는 API/사전 적재 기반 선택형 대시보드, 필요할 때는 업로드형 진단 도구.**

데이터 운영 원칙:
1. **최신성이 중요한 데이터**(인기대출·급상승·키워드·신착)는 도서관정보나루 API 또는 주기적 갱신 데이터를 우선 사용한다.
2. **대용량 장서·대출 원천 데이터**는 모든 공공도서관 것을 한 번에 저장하지 않고, 대표 사례와 최근 분석 결과를 우선 탑재한다.
3. **시연 안정성**을 위해 API 호출 실패 시 저장된 최근 데이터 또는 대표 데이터로 fallback한다.
4. **분석 결과**(수급격차표, 회전율, 추천 후보)는 캐시해 선택형 대시보드가 빠르게 반응하도록 한다.

범위는 우선 **공공도서관**으로 한정한다. 대학도서관·학교도서관·전문도서관은 이용 목적과 장서 운영 기준이 달라 같은 모델로 비교하면 해석이 흔들릴 수 있기 때문이다.

### 작년 대상팀 벤치마킹에서 가져올 것/버릴 것

| 참고 대상 | 가져올 것 | 버릴 것 |
|---|---|---|
| `LIBscope_v2` | 선택형 대시보드, API 우선 구조, 정적 fallback, 캐시/주기적 갱신, 지역 선택에 따른 화면 변화 | 서울시 자치구 중심, 인구·복지·문화시설 지표 과다, 주제와 먼 현황 나열 |
| `LIBanalysis_v2` | `기본 분석 → 심화 분석 → 전략 인사이트` 흐름, 그래프+표 병행, 마지막 전략 요약 | 파일 업로드를 메인으로 두는 구조, 설문 문항 코드에 묶인 분석, GPT 의존적 해석 |

우리식 결론:

> **`LIBscope`의 선택형/API형 구조 + `LIBanalysis`의 기본-심화-전략 흐름을 가져오되, 내용은 장서 수급 진단과 처방으로 바꾼다.**

## 3. 분석의 등뼈 (보고서에 반드시 들어갈 흐름)

```
문제정의 → 데이터 수집·전처리 → EDA → 가설·분석(통계) → 시각화 → 해석·제언
```

1. **가설**: 어떤 도서관이든 장서 구성(공급)과 실제 대출(수요)은 완전히 일치하지 않으며, 그 격차는 장서 운영 의사결정 신호가 된다.
2. **검증**: 표준 장서·대출 데이터에 동일한 진단 엔진을 적용해 세그먼트별 격차와 분포 차이 통계 검정을 수행한다.
3. **결과 → 처방**: 부족 분야 = 수서 추천 / 고회전 단일권 = 복본 추천 / 과잉·저회전 = 정리·큐레이션 후보.

### 세그먼트화 (파생변수)
- **분야**: 주제분류번호(KDC) 맨 앞자리 → 대분류
- **독자대상**: ISBN 부가기호 맨 앞자리 → 교양·실용·청소년·아동·전문

## 4. 코드 설계 (범용 진단 엔진 중심)

> 앱 화면보다 먼저 중요한 것은 **데이터가 어떤 경로로 들어와도 같은 엔진을 통과하는 구조**다.

### 설계 원칙

1. **입력과 분석을 분리한다.**
   - 파일 업로드, 샘플 선택, API 호출은 모두 다른 입구일 뿐이다.
   - 어느 입구로 들어와도 최종적으로는 같은 표준 DataFrame으로 변환한다.

2. **도메인 표준 스키마를 먼저 고정한다.**
   - 필수: `도서권수`, `대출건수`, `주제분류번호`
   - 권장: `ISBN`, `부가기호`, `도서명`, `발행년도`, `등록일자`
   - 파생: `분야`, `독자대상`, `회전율`, `공급비중`, `수요비중`, `격차`

3. **진단 엔진은 Streamlit에 의존하지 않는다.**
   - `diagnosis.py`, `metrics.py`, `prescribe.py`는 화면 없이도 실행 가능해야 한다.
   - 그래야 수업용 노트북, 공모전 데모, 테스트 코드가 같은 로직을 공유한다.

4. **사례 데이터는 갈아끼울 수 있어야 한다.**
   - 대구·금천 파일은 기본 데모일 뿐이다.
   - 새 도서관 파일이 들어와도 `prepare → diagnose → prescribe` 흐름이 그대로 돌아가야 한다.

### 권장 모듈 구조

현재 구현된 구조를 살리되, 역할을 더 명확히 나눈다.

| 모듈 | 책임 | 주요 함수/예시 |
|---|---|---|
| `loader.py` | 파일 업로드·샘플·API/fallback 데이터를 읽어 표준 입력으로 변환 | `smart_read()`, `load_sample_collection()`, `normalize_popular()` |
| `schema.py` *(추가 권장)* | 필수 컬럼 검사, 컬럼 별칭 매핑, 데이터 품질 리포트 | `validate_collection_schema()`, `standardize_columns()` |
| `diagnosis.py` | 장서 데이터를 분야/독자대상으로 세그먼트화하고 수급격차 계산 | `prepare()`, `gap_by_segment()`, `diagnose()` |
| `metrics.py` | 지니계수, 집중도, 통계검정, 회귀용 지표 계산 | `gini()`, `chi_square_gap()`, `turnover_stats()` |
| `prescribe.py` | 진단 결과를 수서·복본·정리 후보로 변환 | `recommend_acquisitions()`, `recommend_duplicates()`, `weeding_candidates()` |
| `api_client.py` *(추가 권장)* | 도서관정보나루 API 호출과 캐시/fallback 관리 | `fetch_popular_books()`, `fetch_library_list()` |
| `app.py` | Streamlit 화면 조립. 분석 로직은 직접 들고 있지 않음 | 도서관 선택, 업로드, 탭 구성, 결과 표시 |

### 표준 실행 흐름

```text
입력 선택
  ├─ 사전 적재 도서관 선택
  ├─ 사용자 파일 업로드
  └─ API/캐시 데이터 호출
        ↓
loader/schema
  - 파일 읽기
  - 컬럼 표준화
  - 필수 컬럼 검사
        ↓
diagnosis
  - KDC → 분야
  - 부가기호 → 독자대상
  - 공급비중/수요비중/격차/회전율 계산
        ↓
metrics
  - 기술통계
  - 지니/집중도
  - 카이제곱 검정
  - 회귀/저회전 분석
        ↓
prescribe
  - 부족 분야 수서 추천
  - 고회전 단일권 복본 추천
  - 과잉·저회전 정리 후보
        ↓
app/report
  - 기본 분석
  - 심화 진단
  - 전략 처방
```

### 화면 설계

작년 대상팀에서 가져올 흐름은 `기본 분석 → 심화 분석 → 전략 인사이트`다. 우리 앱에서는 다음처럼 바꾼다.

| 화면 | 목적 | 보여줄 내용 |
|---|---|---|
| 1. 도서관 선택/업로드 | 어떤 데이터든 같은 엔진으로 들어오게 함 | 사전 적재 도서관 선택, 파일 업로드, 데이터 품질 상태 |
| 2. 기본 분석 | 데이터와 현황 이해 | 장서수, 대출건수, 회전율, 분야/독자대상 분포 |
| 3. 심화 진단 | 모델의 핵심 결과 | 공급비중 vs 수요비중, 격차표, 카이제곱 검정 |
| 4. 전략 처방 | 의사결정으로 연결 | 수서 추천, 복본 추천, 정리·큐레이션 후보 |
| 5. 검증 사례 | 범용성 증명 | 여러 사례에 같은 엔진 적용 결과 비교 |

### 구현 우선순위

1. `schema.py`를 추가해 필수 컬럼 검사와 표준 컬럼 매핑을 분리한다.
2. `diagnosis.py`는 지금처럼 순수 계산 모듈로 유지한다.
3. `prescribe.py`는 추천 결과가 항상 표 형태로 나오게 정리한다.
4. `app.py`는 분석 코드를 직접 쓰지 않고 모듈 결과를 표시만 한다.
5. API 연동은 마지막에 붙인다. 먼저 사전 적재/업로드 데이터로 엔진을 안정화한다.

## 5. 통계 기법을 '질문'에 박기

> *기법을 위한 기법*은 피하고, 각 기법이 질문에서 자연스럽게 나오게 한다. (수업·공모전 둘 다 이걸 좋아함)

| 기법 | 답하는 질문 | 결과물 |
|---|---|---|
| **가설검정(카이제곱)** | 공급 분포 ≠ 수요 분포가 *우연인가?* | 수급격차의 통계적 유의성 = 모델의 근거 |
| **회귀분석** | *무엇이* 대출/회전율을 떨어뜨리나? (발행연도·KDC·독자대상·도서권수) | "안 빌려지는 책의 조건" → 정리·수서 근거 |
| **군집분석** | 장서를 몇 가지 *유형* 으로 묶으면? | 장서 유형 자동 발견 → 처방 그룹화 |
| **기술통계·집중도(지니)** | 쏠림이 얼마나 심한가? | 도서관별 장서·대출 쏠림 정량화 |
| **스키마/품질 점검** | 이 파일은 진단 엔진에 넣어도 되는가? | 필수 컬럼 충족 여부 + 결측/이상치 리포트 |

## 6. 업무분장 (각자 = 분석 질문 1개 + 구현 산출물 1개)

> 코드는 AI 도움으로 작성하되, **각자 자기 단계의 "왜"는 본인이 설명할 수 있어야 한다** (발표·질의 방어용).

| 단계 | 담당 | 맡는 통계 | 산출물(채점 대상) |
|---|---|---|---|
| ① 데이터 운영·스키마·전처리 | 위성 A | 기술통계·품질점검 | 표준 스키마 + 컬럼 매핑표 + 정제 데이터 + API/fallback 데이터 목록 |
| ② EDA·시각화·대시보드 스토리 | 위성 B | 기술통계·군집분석 | 분포·회전율·유형 시각화 + 기본 분석 화면 흐름 |
| ③ 수급격차·가설검정 | 위성 C | 카이제곱 | 격차표 + 유의성 + "엔진이 의미 있는 차이를 잡는가" 해석 |
| ④ 회귀·처방·통합/데모 | **허브(본인)** | 회귀분석 | 저회전 예측 + 수서/복본/정리 추천 + 선택형 대시보드 통합·배포 |

- **허브-위성 구조**: 통합·배포는 웹 경험자(본인) 한 명이 책임 → 충돌 방지. 위성끼리는 의존이 없게 단계를 통째로 배정.
- 위성 누구도 "코드만 돌렸다"가 안 되도록 **분석 단계를 온전히 소유**하게 함.
- 추천 로직(전국 인기대출 ISBN ↔ 자관 미소장 대조)은 처방 파트에 속하므로 ④가 책임지고, ②는 시각화와 사용자 흐름을 중심으로 맡는다.
- ①은 모두의 작업을 막는 선행 병목이 아니다. 공통 컬럼 규격을 먼저 정하고, 나머지 파트는 현재 데이터로 초안을 만든 뒤 최종 정제 데이터로 한 번 갱신한다.
- ③은 "대구와 금천을 비교한다"가 아니라, **같은 엔진이 서로 다른 입력 데이터에서도 통계적으로 의미 있는 진단을 내는지 검증**한다.

---

## 7. 균형 가드레일 (동순위라서 중요)

두 목표가 서로를 잡아먹는 함정 둘:

1. **웹앱 욕심 → 분석 시간 잠식.** 데모는 "돌아가는 수준"에서 멈춘다. 수업 점수는 노트북에서 나온다.
2. **통계 자랑 → 공모전 메시지 매몰.** 제안서엔 p값보다 *"어떤 도서관 데이터가 들어와도 같은 엔진이 수급격차를 계산하고 처방까지 낸다"* 는 한 줄이 먼저 온다.
3. **사례 비교 과몰입 → 주제 축소.** 대구·금천은 검증 사례일 뿐이다. 발표의 주어는 항상 **범용 자동 장서 진단 엔진**이다.

## 8. 다음 액션 (1주)

1. **[엔진 먼저]** 진단 엔진의 표준 입력 스키마를 확정한다. 어떤 도서관 데이터든 이 스키마로 들어오면 돌아가야 한다.
2. **데이터 전략 확정**: API 우선 + 정적 fallback + 캐시 구조로 합의. 업로드는 보조 기능으로 둔다.
3. **위성 A**: 필수 컬럼 검사표와 `schema.py` 초안을 만든다. 도서관정보나루 API로 최신화할 데이터와 사전 적재할 데이터를 분리해 목록화한다.
4. **위성 B**: 도서관 선택형 화면 흐름과 `기본 분석 → 심화 진단 → 전략 처방` 화면 초안 작성.
5. **위성 C**: 검증 사례 데이터에 동일 엔진을 적용해 수급격차표와 카이제곱 검정 초안 작성.
6. **허브**: 전국 인기대출(`BestLoanList`) ISBN ↔ 자관 미소장 대조 = 수서 추천 로직 착수.
7. 팀 회의: 위 분장표 확정 + 수업 제출물 형식(노트북/보고서/발표)에 맞춰 무게중심 조정.
